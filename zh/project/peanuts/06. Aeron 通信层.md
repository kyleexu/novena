# 一、为什么选择 Aeron

传统微服务通过 HTTP/gRPC 通信，每次跨进程调用都需要：
- TCP 连接建立
- 协议序列化/反序列化
- 内核网络栈处理

对于撮合引擎这种**延迟敏感**的场景，这些开销是不可接受的。

**Aeron IPC** 使用**共享内存**（`/dev/shm`）在进程间传递消息，完全绕过内核网络栈，延迟可达**微秒级别**。

---

# 二、核心概念

| 概念 | 说明 |
|------|------|
| MediaDriver | Aeron 的核心进程，管理共享内存缓冲区 |
| Publication | 消息发布方，写入共享内存 |
| Subscription | 消息订阅方，从共享内存读取 |
| Channel | 通信通道，IPC 模式为 `aeron:ipc` |
| Stream ID | 同一 Channel 内的逻辑分流标识，类似 Topic |
| Fragment | 消息分片，大消息会被拆分 |

---

# 三、在 Peanuts 中的用法

### 3.1 MediaDriver 启动

Match Engine 内嵌 MediaDriver，因此**必须第一个启动**：

```yaml
# match/application.yml
match:
  aeron:
    launch-embedded-driver: true
```

其他服务不需要启动 MediaDriver，直接连接 Match 启动的 MediaDriver 即可。

### 3.2 Stream ID 分配约定

```
2001 - 订单入站（Order → Match）
2002 - 执行回报（Match → Order）
2003 - 成交明细（Match → Account, Market）
2004 - 订单簿快照（Match → Market）
2101 - 资金锁定请求（Order → Account）
2102 - 资金锁定响应（Account → Order）
```

# 四、Publisher 和 Subscriber 的抽象封装

系统在 `protocol` 模块中对 Aeron 的使用进行了封装。

### 4.1 AbstractAeronPublisher

所有发布者继承此基类，核心方法是 `offer()`：

```java
public void offer(T message) {
    // 1. 将消息编码为 DirectBuffer（零拷贝）
    // 2. 调用 publication.offer(buffer, 0, length)
    // 3. 处理背压（BACK_PRESSURED / NOT_CONNECTED 等返回值）
}
```

子类只需实现消息的序列化逻辑（Codec）。

### 4.2 AbstractAeronSubscriber

所有订阅者继承此基类，通过 `FragmentHandler` 接收消息：

```java
// 内部轮询循环
public void poll() {
    subscription.poll(fragmentHandler, fragmentLimit);
}
```

`fragmentLimit` 控制每次轮询最多处理多少个消息分片，配置项：

```yaml
match:
  aeron:
    fragment-limit: 10
```

# 五、Codec 编解码层

每个消息类型都有对应的 Codec 负责序列化和反序列化：

| Codec | 消息类型 | Stream |
|-------|----------|--------|
| `OrderCodec` | OrderProto | 2001 |
| `ExecutionReportCodec` | ExecutionReportProto | 2002 |
| `TradeCodec` | TradeProto | 2003 |
| `OrderBookCodec` | OrderBookProto | 2004 |
| `LockRequestCodec` | LockRequestProto | 2101 |
| `LockResponseCodec` | LockResponseProto | 2102 |

编码采用 `UnsafeBuffer`（直接内存），避免 Java 堆 GC 开销。

# 六、请求-响应模式（资金锁定）

Aeron 本身是单向的 pub/sub，但锁定操作需要同步等待结果。系统通过以下方式模拟请求-响应：

```
Order Gateway                    Account Service
     │                                 │
     │─── 2101（LockRequest） ────────►│
     │                                 │ 处理请求
     │◄─── 2102（LockResponse） ────────│
     │
     │ 阻塞等待，最多 timeout-ms（默认 3000ms）
```

Order Gateway 在发出 2101 请求后，在 2102 的 Subscriber 上**轮询等待**响应，超时后判定失败。

这种模式的关键设计：
- LockRequest 需要携带足够的信息让 Account Service 识别请求（userId, currency, amount, orderId）
- LockResponse 需要携带 orderId 与请求对应
- 单线程下不存在并发问题，多线程下需要用 orderId 做请求匹配

# 七、背压处理

Aeron publication 的 `offer()` 可能返回以下负值：

| 返回值 | 含义 |
|--------|------|
| `NOT_CONNECTED` | 没有订阅方连接 |
| `BACK_PRESSURED` | 订阅方消费过慢，发布缓冲区满 |
| `ADMIN_ACTION` | 管理操作中，稍后重试 |
| `CLOSED` | Publication 已关闭 |

系统的处理策略：

```java
long result = publication.offer(buffer, 0, length);
if (result < 0) {
    // 重试或记录错误日志
    log.warn("offer failed: {}", result);
}
```

在低延迟场景下，通常使用**自旋重试**（busy spin）直到发送成功，而不是 sleep。

# 八、与 Kafka 的对比

| 维度 | Aeron IPC | Kafka |
|------|-----------|-------|
| 延迟 | 微秒级 | 毫秒级 |
| 持久化 | 无（共享内存） | 有（磁盘） |
| 跨机器 | 不支持 IPC 模式 | 支持 |
| 消费者组 | 不支持 | 支持 |
| 消息回放 | 不支持 | 支持 |
| 适用场景 | 同机低延迟通信 | 持久化、跨服务、事件溯源 |

Peanuts 的设计是：**核心路径用 Aeron，持久化用 Kafka（bridge 模块）**。

# 九、常见排查点

1. **服务无法连接 MediaDriver**：确认 Match 服务已启动，`/dev/shm/aeron-*` 目录存在
2. **消息丢失**：检查 Subscriber 是否已启动（`subscriber ready` 日志）
3. **背压**：消费者太慢，增大缓冲区或优化消费逻辑
4. **NOT_CONNECTED**：发布者启动时订阅者尚未准备好，需加启动顺序保证
