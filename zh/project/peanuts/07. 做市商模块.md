# 一、职责概述

Maker 模块（`maker` 模块）是一个用于模拟市场流动性的**做市 Bot**，包含两个调度器：

| 调度器 | 角色 | 策略 |
|--------|------|------|
| `MakerLadderScheduler` | 做市商（Maker） | 在买卖两侧按梯形结构持续挂单，维持盘口深度 |
| `TakerFlowScheduler` | 吃单方（Taker） | 定期发送市价/IOC 订单，消耗盘口流动性，产生成交记录 |

Maker 模块连接到外部 WebSocket 行情源获取参考价格，以此为基准确定挂单价格区间。

---

# 二、MakerLadderScheduler：梯形挂单策略

### 2.1 核心概念

**梯形（Ladder）挂单**：在当前市场中间价的两侧，按等比例间距挂出多层订单，形成一个买卖价格梯队，为市场提供流动性。

```
卖单（从低到高）：
  ask level 1: mid × (1 + 3bps)
  ask level 2: mid × (1 + 6bps)
  ask level 3: mid × (1 + 9bps)
  ...
  ask level 10: mid × (1 + 30bps)

买单（从高到低）：
  bid level 1: mid × (1 - 3bps)
  bid level 2: mid × (1 - 6bps)
  ...
  bid level 10: mid × (1 - 30bps)
```

**价格计算公式：**

```
BUY  价格 = mid × (1 - ladderStepBps × level / 10000)
SELL 价格 = mid × (1 + ladderStepBps × level / 10000)
```

### 2.2 主要配置参数

| 参数 | 说明 | 示例值 |
|------|------|--------|
| `ladder-levels` | 挂单层数 | 10 |
| `ladder-step-bps` | 每层价格间距（bps） | 3 |
| `min-ladder-notional-usdt` | 单笔最小名义金额 | 120 |
| `max-ladder-notional-usdt` | 单笔最大名义金额 | 500 |
| `max-active-orders-per-contract` | 每合约最大活跃挂单数 | 50 |
| `min-orders-per-side` | 每侧最少挂单数 | 20 |
| `fixed-delay-ms` | 每次执行间隔（ms） | 1000 |

### 2.3 订单数量计算

挂单数量由名义金额（Notional）除以挂单价格得出：

```
数量 = notional / price
```

靠近中间价的档位名义金额更大（提供更多流动性），远端档位更小，并叠加 ±30% 随机扰动，避免规律性被识别。

### 2.4 Ladder 重建触发条件

以下任一条件满足时，撤销全部挂单并以新中间价重建：

| 条件 | 配置项 | 默认值 |
|------|--------|--------|
| 价格偏离上次建仓中间价 | `rebalance-deviation-bps` | 25 bps |
| 距上次重建超过固定时间 | `rebalance-interval-ms` | 15000 ms |

### 2.5 库存偏置（Inventory Skew）

系统根据当前持仓与目标持仓的差值，自动向买侧或卖侧倾斜挂单概率，实现被动对冲：

```
delta      = 当前 base 持仓 - 目标持仓
normalized = delta / inventorySoftLimitBaseQty   （限制在 [-1, 1]）
bias       = -normalized × inventoryBiasCap
```

- 持仓多 → bias 为负 → 更多卖单（降低持仓）
- 持仓少 → bias 为正 → 更多买单（增加持仓）

# 三、TakerFlowScheduler：吃单流量模拟

### 3.1 作用

Taker 调度器周期性向盘口发送 **市价/IOC 订单**，主动消耗挂单，制造持续的成交流量，用于：

- 测试撮合引擎在持续成交场景下的稳定性
- 为行情服务提供连续的成交数据
- 模拟真实市场的成交节奏

### 3.2 主要配置参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `taker-enabled` | 独立开关 | true |
| `taker-fixed-delay-ms` | 每次执行间隔（ms） | 200 |
| `min-taker-notional-usdt` | 单笔最小吃单金额 | 30 |
| `max-taker-notional-usdt` | 单笔最大吃单金额 | 180 |
| `taker-sweep-bps` | 超出 best ask/bid 的价格冗余（bps） | 1.5 |
| `taker-result-timeout-ms` | 等待回执超时（ms） | 1500 |

### 3.3 Taker 报单策略

```
买入吃单（扫 ask）：
  price = best_ask × (1 + takerSweepBps / 10000)
  确保能穿透盘口买到

卖出吃单（扫 bid）：
  price = best_bid × (1 - takerSweepBps / 10000)
  确保能穿透盘口卖出
```

默认 Taker 频率（200ms）高于 Maker（1000ms），持续消耗 Maker 提供的流动性，形成供需循环。

# 四、外部行情 WebSocket

Maker 通过订阅外部交易所的 WebSocket 获取实时市场中间价，作为挂单基准：

```yaml
maker:
  ticker:
    websocket:
      enabled: true
      urls: wss://stream-pro.hashkey.com/quote/ws/v2
      symbols: BTCUSDT,ETHUSDT
      reconnect-delay-ms: 3000
      heartbeat-interval-ms: 10000
```

**重要逻辑：**

> `TickerCache` 获得第一条价格数据之前，Maker 不会下任何单。

这防止了在参考价格未就绪时以错误价格挂单。

**断线重连：**
- 断开后等待 `reconnect-delay-ms`（默认 3s）再重连
- 支持多个 URL，断连后轮询重试
- 每隔 `heartbeat-interval-ms`（默认 10s）发送心跳 ping

# 五、余额门槛保护

Maker 在选择下单用户时，会检查余额是否充足：

| 条件 | 配置项 | 默认值 |
|------|--------|--------|
| 买单用户 USDT 可用 ≥ | `min-available-quote` | 1000 |
| 卖单用户 BTC 可用 ≥ | `min-available-base` | 0.5 |

余额不足的用户不会被选中下单，避免下单后因资金锁定失败导致流程中断。

# 六、整体流程

```
外部 WebSocket 行情
        │
        ▼
  TickerCache（中间价）
        │
  ┌─────┴────────────────────┐
  │ MakerLadderScheduler     │  每 1000ms
  │  ├── 检查是否需要重建 Ladder
  │  ├── 选择合约和层数
  │  ├── 计算挂单价格和数量
  │  └── 调用 OrderService 下限价单
  └──────────────────────────┘
        │
  ┌─────┴────────────────────┐
  │ TakerFlowScheduler       │  每 200ms
  │  ├── 查询当前盘口
  │  ├── 计算吃单价格
  │  └── 调用 OrderService 下市价/IOC 单
  └──────────────────────────┘
        │
        ▼
   Order Gateway → Account → Match Engine → 成交
```
