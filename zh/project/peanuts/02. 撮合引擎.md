# 一、职责概述

Match Engine 是整个系统的核心，负责：

1. 维护每个合约的**内存订单簿**
2. 执行**价格优先、时间优先**的撮合算法
3. 处理新订单、改单、撤单三种操作
4. 撮合完成后发布**成交记录**和**执行回报**

Match 服务没有 HTTP 端口，只通过 Aeron IPC 与外界通信。它同时内嵌 Aeron MediaDriver，因此必须**第一个启动**。

---

# 二、订单簿数据结构

订单簿（`OrderBook.java`）由三个数据结构组成：

```java
// 买单队列：按价格降序，同价按时间升序（最优先成交价格最高、到达最早的买单）
PriorityQueue<Order> buyOrders = new PriorityQueue<>(comparator);

// 卖单队列：按价格升序，同价按时间升序（最优先成交价格最低、到达最早的卖单）
PriorityQueue<Order> sellOrders = new PriorityQueue<>(comparator);

// ID 索引：用于 O(1) 查找（改单/撤单）
Map<Long, Order> ordersById = new HashMap<>();
```

**买单排序规则（最大堆）**：

```java
// 价格高的排前面，同价按时间升序（早来的优先）
int priceCompare = right.getPrice().compareTo(left.getPrice());
if (priceCompare != 0) return priceCompare;
return Long.compare(left.getTimestamp(), right.getTimestamp());
```

**卖单排序规则（最小堆）**：

```java
// 价格低的排前面，同价按时间升序
int priceCompare = left.getPrice().compareTo(right.getPrice());
if (priceCompare != 0) return priceCompare;
return Long.compare(left.getTimestamp(), right.getTimestamp());
```

系统为每个合约（BTC_USDT、ETH_USDT、USDT_USD）维护独立的 OrderBook 实例：

```java
Map<Contract, OrderBook> books = new HashMap<>();
```

# 三、撮合核心算法

`MatchService.match()` 是整个引擎的入口，所有操作都在 `synchronized` 块内执行（单线程保证数据一致性）。

## 3.1 订单分发入口

```
match(order)
  ├── normalize(order)        // 补全默认值
  ├── if CANCEL → cancel()    // 直接撤单
  ├── if MODIFY → cancelExisting() + 重建订单 + newOrder()
  └── else → newOrder()       // 正常撮合
```

## 3.2 新订单撮合流程（newOrder）

```
1. 取对向队列（BUY → sellOrders，SELL → buyOrders）
2. 循环尝试撮合：
   a. 取对向队列队首（最优价格）
   b. isCrossed()：检查价格是否交叉
      - 市价单：始终交叉（true）
      - 限价买单：incomingPrice >= restingPrice
      - 限价卖单：incomingPrice <= restingPrice
   c. 成交数量 = min(remaining(incoming), remaining(resting))
   d. 成交价格 = resting 订单价格（以挂单价成交）
   e. 更新双方已成交数量
   f. 发布 Trade + 两条 ExecutionReport
   g. 若对向订单完全成交，从队列中移除
3. 循环结束后处理剩余：
   a. shouldRestOnBook()：MARKET 单和 IOC/FOK 不进入订单簿
   b. 可挂单：加入订单簿
   c. 不可挂单：根据是否有成交设置 PARTIALLY_FILLED 或 CANCELED
```

## 3.3 价格交叉判断

```java
private boolean isCrossed(Order incoming, Order resting) {
    if (incoming.getOrderType() == OrderType.MARKET) return true;
    if (incoming.getSide() == Side.BUY)
        return incoming.getPrice().compareTo(resting.getPrice()) >= 0;
    return incoming.getPrice().compareTo(resting.getPrice()) <= 0;
}
```

## 3.4 是否挂单判断

```java
private boolean shouldRestOnBook(Order order) {
    if (order.getOrderType() == OrderType.MARKET) return false;
    return order.getTimeInForce() != TimeInForce.IOC
        && order.getTimeInForce() != TimeInForce.FOK;
}
```

# 四、支持的订单类型与时效性

| 订单类型 | TimeInForce | 行为 |
|----------|-------------|------|
| LIMIT | GTC | 未成交部分挂入订单簿，等待下次撮合 |
| LIMIT | IOC | 立即成交，剩余部分直接取消 |
| LIMIT | FOK | 必须全部成交，否则全部取消 |
| MARKET | — | 以对手价立即成交，不挂单 |

# 五、改单流程

改单（MODIFY）采用 **Cancel + Re-Add** 模式：

```
1. 解析 targetOrderId（目标订单）
2. 从订单簿中移除目标订单
3. 复用目标订单的 orderId
4. 重置 filledQuantity = 0，status = NEW，action = NEW
5. 执行 newOrder()，重新撮合
```

> 改单时间戳会更新，因此在同价位时优先级会排到队尾。

---

# 六、撤单流程

```
1. 解析 targetOrderId
2. 从 ordersById 中移除
3. 从对应的 buyOrders 或 sellOrders 中移除
4. 发布 CANCELED 状态的 ExecutionReport
```

# 七、成交后的消息发布

每次成交触发以下两种消息的发布：

### TradeProto（Stream 2003）

```java
trade.setTradeId(tradeIdGenerator.incrementAndGet());  // 基于 System.currentTimeMillis() * 1000
trade.setBuyOrderId / setSellOrderId / setBuyUserId / setSellUserId
trade.setPrice(matchedPrice);     // 以挂单价成交
trade.setQuantity(matchedQuantity);
trade.setTimestamp(System.currentTimeMillis());
```

### ExecutionReportProto（Stream 2002）

每次成交发布**两条** ExecutionReport：一条给买方订单，一条给卖方订单，tradeId 相同。

执行类型（ExecutionType）包括：
- `TRADE`：部分或全部成交
- `CANCELED`：撤单
- `REJECTED`：拒绝

# 八、线程模型

`MatchService` 的 `match()` 方法是 `synchronized` 的，整个撮合是**单线程串行**执行。这是撮合引擎的常见设计：

- 避免并发导致订单状态不一致
- 订单簿无需加锁，操作开销极低
- Aeron 订阅方（`OrderSubscriber`）从 Poller 线程调用 `match()`，天然串行

---

# 九、TradeId 生成策略

```java
AtomicLong tradeIdGenerator = new AtomicLong(System.currentTimeMillis() * 1_000L);
```

以毫秒时间戳乘以 1000 作为起始值，每次成交自增 1。这样：
- 天然有序（大体按时间递增）
- 重启后不会与历史 tradeId 重复（只要不在同一毫秒重启）
- 不依赖数据库序列或分布式 ID 生成器
