# 一、职责概述

Account Service（`account` 模块，端口 8081）负责：

1. 维护用户各货币的**内存余额**（可用 + 冻结）
2. 响应订单网关的资金锁定请求
3. 接收撮合成交消息，完成资金结算
4. 提供 HTTP 余额查询与操作接口

---

# 二、账户数据模型

每个账户快照（`AccountAssetSnapshot`）包含：

```
userId + currency → { available: BigDecimal, locked: BigDecimal }
```

**货币枚举（Currency）：**
- `BTC`
- `ETH`
- `USDT`
- `USD`

**合约与货币关系：**

| 合约 | 基础货币（base） | 计价货币（quote） |
|------|-----------------|-----------------|
| BTC_USDT | BTC | USDT |
| ETH_USDT | ETH | USDT |
| USDT_USD | USDT | USD |

---

# 三、余额操作接口

`AccountService` 提供以下原子操作：

| 方法 | 说明 |
|------|------|
| `query(userId, currency)` | 查询账户快照 |
| `increaseAvailable(userId, currency, amount)` | 增加可用余额（充值） |
| `tryLock(userId, currency, amount)` | 将可用余额转为冻结（下单锁仓） |
| `deductAvailable(userId, currency, amount)` | 直接扣减可用余额 |
| `deductLocked(userId, currency, amount)` | 扣减冻结余额（结算扣款） |
| `transferIncrease(fromUserId, toUserId, currency, amount)` | 两个用户间转账（结算增款） |
| `applyTrade(trade)` | 根据成交记录自动结算 |

# 四、资金锁定流程（下单时）

当订单网关发来锁定请求（Stream 2101）时，账户服务执行：

```
1. 检查对应货币的可用余额 >= 锁定金额
2. available -= amount
3. locked += amount
4. 返回 LockResponse(success=true)
```

若余额不足：

```
4. 返回 LockResponse(success=false, message="insufficient balance")
```

# 五、成交结算流程（撮合后）

账户服务订阅 Stream 2003（TradeProto），接收每笔成交后调用 `applyTrade(trade)`：

```java
public boolean applyTrade(Trade trade) {
    return accountCache.settleTrade(
        trade.getBuyUserId(),
        trade.getSellUserId(),
        trade.getContract().getBase(),   // 基础货币
        trade.getContract().getQuote(),  // 计价货币
        trade.getPrice(),
        trade.getQuantity()
    );
}
```

**结算逻辑（以 BTC_USDT 成交为例）：**

```
买方：
  - locked(USDT) -= price × quantity   （扣减已锁定的 USDT）
  - available(BTC) += quantity          （获得 BTC）

卖方：
  - locked(BTC) -= quantity             （扣减已锁定的 BTC）
  - available(USDT) += price × quantity  （获得 USDT）
```

# 六、HTTP API

### 6.1 查询余额

```
GET /api/accounts/{userId}/currencies/{currency}
```

响应示例：

```json
{
  "userId": 1001,
  "currency": "BTC",
  "available": "0.50000000",
  "locked": "0.10000000"
}
```

### 6.2 充值（增加可用余额）

```
POST /api/accounts/increase
{
  "userId": 1001,
  "currency": "USDT",
  "amount": 10000.00
}
```

### 6.3 锁定余额

```
POST /api/accounts/lock
{
  "userId": 1001,
  "currency": "USDT",
  "amount": 4500.00
}
```

### 6.4 扣减可用余额

```
POST /api/accounts/deduct/available
```

### 6.5 扣减冻结余额

```
POST /api/accounts/deduct/locked
```

### 6.6 转账

```
POST /api/accounts/transfer-increase
{
  "fromUserId": 1001,
  "toUserId": 1002,
  "currency": "USDT",
  "amount": 100.00
}
```

# 七、Aeron 配置

账户服务同时作为**订阅方**和**发布方**：

```yaml
account:
  aeron:
    channel: aeron:ipc
    lock-request-stream-id: 2101    # 订阅：接收锁定请求
    lock-response-stream-id: 2102   # 发布：返回锁定结果
    trade-stream-id: 2003           # 订阅：接收成交记录
```

# 八、设计要点

1. **纯内存存储**：当前实现没有数据库持久化，重启后账户数据丢失。生产环境需要持久化到数据库或通过 Raft 日志回放重建。

2. **锁定前置于撮合**：资金在订单进入撮合引擎之前就已锁定，确保成交后结算一定能成功。

3. **无分布式事务**：锁定（2101/2102）和结算（2003）是两个独立的 Aeron 流，通过业务逻辑保证一致性，而非分布式事务。

4. **非线程安全警告**：`AccountCache` 直接操作 HashMap 和 BigDecimal，在当前单线程 Aeron Poller 模型下是安全的，但如果引入并发访问需要加锁。
