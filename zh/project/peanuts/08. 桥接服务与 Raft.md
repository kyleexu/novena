# 一、职责概述

Bridge 模块（`bridge`）负责将 Aeron IPC 消息流持久化到 Kafka，同时通过 **Raft 共识协议**保证消息在多节点之间的一致性复制。

**设计目标：** 核心路径（撮合、账户）走低延迟的 Aeron IPC；持久化、审计、下游消费走 Kafka。Bridge 是这两层之间的桥梁。

---

# 三、为什么需要 Raft？

Aeron IPC 本身是**不可靠的**（消息可能因订阅者未就绪而丢失），而 Kafka 要求**精确一次**或**至少一次**的语义。

引入 Raft 共识的好处：
1. 消息先提交到 Raft 日志（多数节点确认），再写入 Kafka，避免消息丢失
2. 节点崩溃后可以从 Raft 日志中恢复，保证消息不重不漏
3. 单节点部署时，Raft 退化为本地日志，依然保证顺序

# 四、Raft 完整流程

以 Stream 2003（成交记录）为例，一条消息从 Aeron 到 Kafka 的完整路径：

```
Step 1. Aeron Poller 收到消息
        AbstractAeronSubscriber.handleMessage()

Step 2. 提案到 Raft Leader
        RaftBootstrap.apply(payload)
          └── Node#apply(task)

Step 3. Raft 协议：复制到 Follower（网络）
        Leader → 多数 Follower 写入日志
                  多数确认 → commit

Step 4. 各节点状态机触发 onApply
        CodecRaftStateMachine.onApply(iter)
          └── codec.decode()
          └── applyHandler.onCommitted(message, ...)

Step 5. Bridge 写入 Kafka
        RaftKafkaMessageApplyHandler.onCommitted()
          └── objectMapper.writeValueAsString(message)
          └── kafkaTemplate.send(topic, json)
```

---

# 五、两种 Raft 应用模式

`AbstractAeronSubscriber` 支持两种消息处理时机，通过 `raft-apply-mode` 配置：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `ON_AERON_POLL` | 消息从 Aeron 收到时**立即**处理（提案入 Raft，但 onMessage 在 Leader 提案时触发） | 低延迟，不要求严格持久化 |
| `AFTER_COMMIT` | 等 Raft commit（多数节点确认）后**再**调用 `onMessage` | 强一致，要求不丢消息 |

Bridge 模块使用 `AFTER_COMMIT` 模式：只有 Raft 多数节点确认后，才将消息发送到 Kafka。

# 六、状态机核心代码

```java
// CodecRaftStateMachine.java
@Override
public void onApply(Iterator iter) {
    while (iter.hasNext()) {
        ByteBuffer data = iter.getData();
        byte[] bytes = new byte[data.remaining()];
        data.get(bytes);

        // 反序列化消息
        M message = codec.decode(new UnsafeBuffer(bytes), 0);

        Closure done = iter.done();
        // 触发业务处理（Bridge 场景：写 Kafka）
        applyHandler.onCommitted(message, done != null, done);
        iter.next();
    }
}
```

```java
// RaftKafkaMessageApplyHandler.java
@Override
public void onCommitted(T message, boolean localApply, Closure done) {
    // 序列化为 JSON
    String json = objectMapper.writeValueAsString(message);

    // 写入 Kafka（支持可选的消息 Key）
    String key = keyExtractor == null ? null : keyExtractor.apply(message);
    if (key == null) {
        kafkaTemplate.send(topic, json);
    } else {
        kafkaTemplate.send(topic, key, json);
    }

    if (localApply && done != null) {
        done.run(Status.OK());
    }
}
```

# 七、Raft 配置

每个 Stream 对应一个独立的 Raft 组（单节点模式）：

```yaml
# bridge/application.yml
bridge:
  streams:
    - streamId: 2002
      topic: peanuts-execution-report
      raft:
        dataPath: /tmp/peanuts-raft/bridge/stream-2002
        groupId: peanuts-bridge-stream-2002
        serverId: 127.0.0.1:7002
        initConf: 127.0.0.1:7002

    - streamId: 2003
      topic: peanuts-trade
      raft:
        dataPath: /tmp/peanuts-raft/bridge/stream-2003
        groupId: peanuts-bridge-stream-2003
        serverId: 127.0.0.1:7003
        initConf: 127.0.0.1:7003
```

**配置项说明：**

| 配置项 | 说明 |
|--------|------|
| `streamId` | 对应的 Aeron Stream ID |
| `topic` | 目标 Kafka Topic |
| `dataPath` | Raft 日志存储路径 |
| `groupId` | Raft 组 ID（唯一标识） |
| `serverId` | 当前节点地址（`IP:PORT`） |
| `initConf` | 初始集群配置（单节点时与 serverId 相同） |

---

# 八、基础设施依赖

Bridge 模块是唯一依赖外部基础设施的模块：

| 依赖 | 地址 | 说明 |
|------|------|------|
| Kafka | localhost:9092 | 消息写入目标 |
| jRaft | 各 Raft 端口 | 共识协议（进程内启动，无需单独部署） |

其他模块（order、account、market、match）**无需** Kafka 和 Raft，纯 Aeron IPC 通信。

---

# 九、单节点 vs 多节点

当前配置为**单节点 Raft**（`serverId == initConf`），等价于顺序写本地日志，没有网络复制的开销，但也失去了多节点高可用的保障。

扩展为多节点时，修改：
```yaml
raft:
  serverId: 127.0.0.1:7002
  initConf: 127.0.0.1:7002,127.0.0.2:7002,127.0.0.3:7002
```

并在三个节点上分别启动 Bridge 服务，Raft 会自动选举 Leader 并复制日志。
